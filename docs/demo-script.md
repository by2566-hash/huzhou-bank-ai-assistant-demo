# 国内银行 App 可信 AI 任务助手 Demo 演示脚本

## Demo 目标
用一段手机银行原型演示：普通用户不需要熟悉银行菜单和术语，只要说出目标，AI 助手就能在安全验证后返回可直接使用的任务卡片。

第一版演示重点：

- 目标用户是普通个人、长辈用户和低频关键任务用户，不是企业财务或专业汇款用户。
- 所有数据均为 fictional mock data。
- 第一版只做只读查询和信息生成，不执行真实转账、挂失、导出或分享。
- 便利和安全同时展示：能直接给结果，但敏感信息必须验证、脱敏、可审计。

## 演示设定
- Demo 银行语境：湖州银行。
- Demo 用户：林晨，fictional personal user。
- 初始状态：展示一次人脸认证登录画面，让观众明确用户已经完成登录并处于银行 App 登录态。
- 二次验证：用 mock Face ID / 支付密码弹层表示。
- 数据来源：账户资料、网点资料、交易明细、扣款协议、权限策略均为 mock source data。
- 所有账号、支行、联行号、交易记录和授权 App 都是演示数据，不代表真实银行数据或真实系统接入。

## 开场讲法
传统手机银行的问题不是没有功能，而是普通用户不知道该去哪里找、字段是什么意思、哪些信息可以安全分享。这个 demo 展示一个可信 AI 任务助手：用户说目标，系统判断风险等级，必要时验证身份，然后直接生成结构化、可复制、可分享、可追溯的结果卡片。

## Demo 页面顺序
0. 人脸认证登录页：展示 mock Face ID 登录成功，进入手机银行首页。
1. 手机银行首页：顶部放置贯穿整行的 AI 搜索栏，作为主入口。
2. 搜索栏轮播提示：每 3 秒只在两句淡色 placeholder 之间切换：“你好，我是小X，有什么可以帮到您？”和“请直接复制你的问题”。
3. AI 助手首页：显示输入框、语音入口、快捷问题。
4. 账户/银行卡选择弹窗：当问题对应多张银行卡、多个账户或多个结果时出现。
5. 身份验证弹层：用于银行卡号、完整账号、完整明细、分享/导出动作。
6. 结果卡片：按场景显示用户所问内容，保持精简。
7. 渐进式信任提示：卡片只保留轻量来源/更新时间；敏感操作时再显示验证、隐私或审计说明。

## 交互原则
- 先答用户问的问题，不主动塞入用户没问的附加信息。
- 卡片内容保持短而可操作；更多字段放到“展开详情”或后续追问。
- 如果用户的问题可能对应多张银行卡、多个账户、多个交易对象、多个扣款协议，先弹窗让用户选择。
- 涉及银行卡号、完整账号、完整明细、扣款协议、分享或导出时，先进行隐私验证。
- 审计和隐私提示不应抢占主要视觉；使用小图标、轻量文案、确认弹窗和“查看来源”承载。

## 快捷问题建议
AI 助手首页可显示这些快捷问题：

- 公司报销要收款信息
- 我的银行卡号是多少
- 我的开户行和联行号
- 查给某人的转账
- 这笔扣款是什么
- 哪些 App 正在自动扣我的银行卡
- 最近一个月消费总结
- 银行卡丢了怎么办
- 我的转账限额是多少

## 场景 1：境内收款信息生成

**用户痛点：**  
公司报销、学校退款、平台打款或朋友转账时，用户常被要求填写户名、账号、开户行、联行号等字段。普通用户不知道这些字段在哪里，也容易复制错。

**用户输入：**
> 公司报销要我的收款信息，我要提供什么？

**风险等级：** 高敏。涉及账户信息、完整账号、可分享收款信息单。

**系统动作：**
1. 识别 intent：domestic_receipt_info。
2. 回复一句短提示：  
   “这会展示你的账户收款信息。为保护账户安全，需要先验证身份。”
3. 弹出 mock Face ID / 支付密码验证。
4. 如果用户有多张银行卡，弹出银行卡选择：
   - 湖州银行借记卡 尾号 8026
   - 湖州银行工资卡 尾号 3391
5. 用户选择后显示境内收款信息卡片。

**结果卡片字段：**
- 户名：林晨
- 银行名称：湖州银行
- 账户类型：借记卡
- 账号：**** **** **** 8026
- 开户支行：湖州银行演示支行
- 开户地：浙江省湖州市
- 联行号/CNAPS：3132900XXXXX

默认卡片只展示完成“报销收款信息”必需字段。数据来源和更新时间作为轻量说明或“查看来源”入口，不用大段文字占据主内容。

**卡片按钮：**
- 复制全部
- 生成收款信息单
- 分享
- 显示完整账号
- 查看数据来源

**单行操作：**
每一行都提供独立复制按钮：
- 复制户名
- 复制银行名称
- 复制账号
- 复制开户支行
- 复制开户地
- 复制联行号

**显示完整账号交互：**
1. 用户点击“显示完整账号”。
2. 系统提示：  
   “完整账号属于敏感信息。本次查看会记录在安全日志中。”
3. 用户确认后显示完整 mock 账号。

**高管讲解点：**
这个场景展示的是“结果直接可用”。用户不用分别查账户、开户行、联行号，也不用截图整理。整卡复制适合报销系统，单行复制适合对方只要求某一个字段。

**成功标准：**
- 一个自然语言问题。
- 一次身份验证。
- 一张可直接复制的收款信息卡片。
- 敏感字段默认脱敏，完整展示有确认和审计提示。

## 场景 2：银行卡号查询

**用户痛点：**  
普通用户经常只需要复制自己的银行卡号，但 App 里可能要先进账户、点卡片、再点详情。多张卡时还容易复制错。

**用户输入：**
> 我的银行卡号是多少？

**风险等级：** 高敏。涉及完整银行卡号。

**系统动作：**
1. 识别 intent：card_number_lookup。
2. 提示需要验证身份。
3. 通过 mock Face ID / 支付密码验证。
4. 如果有多张银行卡，弹出银行卡选择：
   - 湖州银行借记卡 尾号 8026
   - 湖州银行工资卡 尾号 3391
5. 用户选择后显示银行卡号卡片。

**结果卡片字段：**
- 银行卡：湖州银行工资卡 尾号 3391
- 卡号：**** **** **** 3391

默认只显示用户问的银行卡号，不附带开户行、网点、限额等额外信息。

**卡片按钮：**
- 复制卡号
- 显示完整卡号
- 换一张卡

**显示完整卡号交互：**
用户点击“显示完整卡号”后，出现轻量确认：
> 完整卡号属于敏感信息，请确认周围环境安全。

**高管讲解点：**
这个场景很小，但代表普通用户真实痛点：用户不是要学习账户管理，只是要快速、安全地复制卡号。

**成功标准：**
- 多卡时先选择，不替用户猜。
- 完整卡号默认隐藏。
- 复制卡号路径短。

## 场景 3：开户行 / 网点 / 联行号直达查询

**用户痛点：**  
很多用户知道银行卡号，但不知道开户行和联行号。传统 App 往往让用户进入账户管理、网点查询、在线客服或公众号，多次跳转后才能找到结果。

**用户输入：**
> 我的开户行和联行号是什么？

**风险等级：** 中敏。可展示支行和联行号，但账号默认只显示尾号。

**系统动作：**
1. 识别 intent：branch_cnaps_lookup。
2. 如果有多张银行卡，先弹出银行卡选择。
3. 判断当前登录态足够展示支行与联行号。
4. 直接显示开户行卡片，不返回菜单路径。

**结果卡片字段：**
- 账户：尾号 8026 借记卡
- 开户行：湖州银行演示支行
- 联行号/CNAPS：3132900XXXXX

默认只回答用户问的“开户行和联行号”。网点地址、电话、适用场景放入“展开详情”，避免卡片信息过多。

**卡片按钮：**
- 复制开户行
- 复制联行号
- 复制全部
- 展开详情
- 解释“联行号是什么”

**长辈友好解释：**
用户点击“解释联行号是什么”后，助手显示：
> 联行号可以理解为银行网点在跨行转账系统里的编号。对方让你填联行号时，通常就是为了确认钱转到正确的开户网点。

**高管讲解点：**
这个场景不是复杂 AI，而是高频、低复杂度、高烦恼的体验优化。用户不需要学习银行菜单，直接拿到字段。

**成功标准：**
- 不只返回“请到某菜单查询”。
- 直接给出开户行和联行号。
- 提供单字段复制和普通语言解释。

## 场景 4：对某人的转账 / 消费记录查询

**用户痛点：**  
普通用户记忆交易时通常按人、商户和场景记，不按银行筛选字段记。例如“我给王明转过几次”“上个月医院花了多少钱”。

**用户输入：**
> 查一下我最近给王明的转账。

**风险等级：** 中高敏。汇总可展示，完整明细、导出、分享需要二次验证或审计提示。

**系统动作：**
1. 识别 intent：counterparty_transaction_search。
2. 匹配 counterparty：王明。
3. 默认时间范围：最近 90 天。
4. 返回汇总卡片和脱敏明细。

**结果卡片字段：**
- 查询对象：王明
- 时间范围：最近 90 天
- 匹配交易：3 笔
- 总金额：2,460.00 元
- 最近一笔：2026-05-21，800.00 元，转账成功
- 数据来源：交易明细
- 更新时间：2026-06-01 10:31

**明细展示：**
- 2026-05-21 800.00 元 转账成功
- 2026-04-08 860.00 元 转账成功
- 2026-03-02 800.00 元 转账成功

**卡片按钮：**
- 只看 5 月
- 查看完整明细
- 复制摘要
- 查看电子回单
- 导出记录

**安全交互：**
- 点击“查看完整明细”触发 mock 二次验证。
- 点击“导出记录”显示提示：  
  “交易记录包含敏感金融信息，导出会记录在安全日志中。”

**高管讲解点：**
这展示了用户自然语言查询交易的价值。不是让用户在明细页里手动筛选，而是用人类记忆方式查银行记录。

**成功标准：**
- 支持按人/商户/关键词查交易。
- 默认先给汇总，避免过度暴露明细。
- 完整明细和导出有更强安全处理。

## 场景 5：近期消费总结

**用户痛点：**  
银行 App 有交易明细，但普通用户想知道的是钱花在哪、有没有重复扣费、哪些是自动扣款，而不是逐条看流水。

**用户输入：**
> 总结一下我最近一个月消费。

**风险等级：** 中敏。展示汇总和分类，不展示完整账号。继续查看具体明细时提高权限要求。

**系统动作：**
1. 识别 intent：monthly_spending_summary。
2. 读取 mock 交易数据。
3. 生成消费洞察卡片。

**结果卡片字段：**
- 时间范围：2026-05-01 至 2026-05-31
- 总支出：6,842.50 元
- 最大类别：餐饮 2,180.00 元
- 高频商户：美团、盒马、地铁出行
- 固定扣费：视频会员 29.00 元，云存储 12.00 元
- 可能重复扣款：5 月 18 日同一商户两笔 68.00 元
- 数据来源：交易明细分类
- 更新时间：2026-06-01 10:32

**卡片按钮：**
- 查看餐饮明细
- 查看固定扣费
- 查看疑似重复扣款
- 设置扣款提醒
- 复制消费摘要

**高管讲解点：**
这个场景说明银行 App 不只是被动记录交易，也可以帮用户理解资金流向。它适合普通人，也适合长辈用户发现异常扣款。

**成功标准：**
- 不输出泛泛理财建议。
- 只基于 mock 交易数据总结。
- 对异常扣款提供“查看详情/联系商户/安全中心”等安全下一步。

## 场景 6：这笔扣款是什么

**用户痛点：**  
普通用户看到一笔不熟悉的扣款时，最想知道“这是谁扣的、是不是我授权过、要不要处理”。传统明细页常只有商户名和金额，解释不够直观。

**用户输入：**
> 这笔扣款是什么？

**风险等级：** 中敏。解释单笔交易可在登录态展示；进一步投诉、冻结、导出或查看完整授权信息需要更强验证。

**系统动作：**
1. 识别 intent：charge_explanation。
2. 如果用户没有指定交易，弹出最近交易选择列表。
3. 用户选择一笔交易后，显示扣款解释卡片。

**结果卡片字段：**
- 扣款商户：云听会员
- 金额：29.00 元
- 时间：2026-05-28 08:12
- 扣款卡：尾号 8026
- 说明：这看起来是一笔会员自动续费。

默认只解释用户问的这一笔，不主动展开全部历史扣款。

**卡片按钮：**
- 查看同类扣款
- 查看自动扣款协议
- 我不认识这笔
- 联系商户

**安全交互：**
- 点击“查看自动扣款协议”前，如果涉及完整授权信息，触发 mock 二次验证。
- 点击“我不认识这笔”进入安全中心路径，不直接替用户发起争议或冻结。

**高管讲解点：**
这个场景非常贴近普通用户和长辈用户。AI 的价值不是泛泛总结，而是把一笔用户看不懂的交易解释清楚，并给出安全下一步。

**成功标准：**
- 只解释用户选择的那笔扣款。
- 不把整月消费和所有历史交易塞进卡片。
- 对异常处理给官方路径，不直接执行高风险动作。

## 场景 7：哪些 App 正在自动扣我的银行卡

**用户痛点：**  
用户常常不知道哪些 App 或服务绑定了银行卡、哪些会自动扣款。传统 App 里入口分散，长辈用户尤其难找。

**用户输入：**
> 哪些 App 正在自动扣我的银行卡？

**风险等级：** 高敏。涉及银行卡、授权协议和交易习惯。

**系统动作：**
1. 识别 intent：recurring_payment_authorizations。
2. 触发 mock Face ID / 支付密码验证。
3. 如果有多张银行卡，弹出银行卡选择。
4. 用户选择后，显示该卡相关自动扣款卡片。

**结果卡片字段：**
- 银行卡：湖州银行借记卡 尾号 8026
- 自动扣款数量：3 个
- 云听会员：每月 29.00 元，最近扣款 2026-05-28
- 视频会员：每月 39.00 元，最近扣款 2026-05-16
- 云存储：每月 12.00 元，最近扣款 2026-05-03

默认只展示用户选择银行卡上的自动扣款摘要。取消授权、查看完整协议、导出清单进入官方流程或二次验证。

**卡片按钮：**
- 管理云听会员
- 查看全部协议
- 换一张卡
- 设置扣款提醒

**高管讲解点：**
这是一个很适合普通用户的 AI 场景：银行已有扣款和授权数据，但用户很难自己整理。AI 助手可以把分散入口变成一个清晰列表。

**成功标准：**
- 多卡时必须先选择银行卡。
- 不直接取消授权，只进入官方管理流程。
- 展示内容简洁，默认不展开协议全文。

## 场景 8：银行卡丢失前置指引

**用户痛点：**  
银行卡丢失是低频但高焦虑场景。长辈用户尤其容易不知道先冻结、挂失、补卡还是联系客服。

**用户输入：**
> 银行卡丢了怎么办？

**风险等级：** 高风险入口，但第一版只做指引，不执行真实挂失。

**系统动作：**
1. 识别 intent：lost_card_guidance。
2. 给出清晰说明，不执行真实操作。
3. 引导用户进入官方挂失流程。

**结果卡片字段：**
- 建议第一步：先临时冻结或挂失，避免资金风险。
- 冻结和挂失区别：
  - 临时冻结：适合不确定是否丢失，可后续恢复。
  - 正式挂失：适合确认丢失，通常需要补卡。
- 可能影响：取现、刷卡、线下支付、部分自动扣款。
- 下一步：进入官方挂失页面。
- 安全提示：任何挂失/解挂/补卡都需要二次验证。

**卡片按钮：**
- 进入挂失页面
- 先冻结卡片
- 联系客服
- 查看附近网点

**高管讲解点：**
这个场景体现 AI 助手不只是查信息，还能在用户焦虑时给清晰、安全、低误操作的路径。但第一版不替用户执行高风险操作。

**成功标准：**
- 说明清楚冻结和挂失区别。
- 不模拟真实资金或卡状态变更。
- 高风险动作全部导向官方流程和二次验证。

## 备选快捷场景
这些不作为第一版主演示，但可放在助手首页快捷问题或后续版本说明中：

- 我的转账限额是多少？
- 为什么我今天转不了账？
- 这个月工资到账了吗？
- 上周给房东的转账凭证在哪？
- 我要办签证，怎么开银行流水？

## 演示闭环讲法
演示结束时强调：

1. 它不是普通客服问答，而是银行任务入口。
2. 它解决普通用户低频但关键任务的困扰。
3. 它把菜单跳转、术语理解、字段复制、敏感信息保护合并成一个可控流程。
4. 第一阶段只读查询，风险可控。
5. 未来可以接入真实账户服务、网点服务、交易服务、权限策略、审计日志和评测系统。

## 原型验收要点
后续做前端时，至少要能演示：

- 首屏有人脸认证登录状态，让观众知道用户已处于登录态。
- AI 助手入口明显。
- 首页顶部有贯穿整行的 AI 搜索栏，并支持 3 秒轮播 placeholder。
- 用户可以点击快捷问题或输入自然语言。
- 银行卡号、收款信息、自动扣款协议等敏感问题至少显示一次 mock 身份验证。
- 多张银行卡或多个匹配结果时，必须出现选择弹窗。
- 境内收款信息卡片支持整卡复制和单行复制。
- 开户行卡片提供“解释联行号是什么”。
- 交易查询默认展示汇总，完整明细需要更强验证。
- 消费总结不做投资建议，只做交易理解。
- “这笔扣款是什么”只解释用户选中的扣款，不默认展开过多信息。
- “哪些 App 正在自动扣我的银行卡”先选卡，再展示授权摘要。
- 挂失场景只做安全指引，不执行真实高风险动作。
- 来源、更新时间、隐私提示采用轻量或按需展开方式，不喧宾夺主。
