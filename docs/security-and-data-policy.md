# 安全与数据策略

## 目标
为国内银行 App 可信 AI 任务助手 demo 定义实现前的安全边界。这个策略服务两个目标：

- 对普通用户：让银行任务更直接、更少跳转、更容易理解。
- 对银行侧：确保账户、交易、授权协议和敏感操作不会被 AI 随意展示、编造或执行。

第一版是高保真原型和 mock 数据 demo，不接真实银行系统，不处理真实账户，不执行真实交易、挂失、导出或授权取消。

## 基本原则
1. **只读优先**：第一版只做查询、解释、整理和官方流程指引，不执行资金、卡状态、授权协议的真实变更。
2. **用户问什么先答什么**：结果卡片只展示完成当前问题所需字段，额外信息放到展开、查看来源或后续追问。
3. **敏感信息默认隐藏**：完整卡号、完整账号、完整交易明细、扣款协议详情默认不直接展示。
4. **多对象先选择**：多张银行卡、多个账户、多个交易对象、多个扣款协议时，先让用户选择，不替用户猜。
5. **验证前置**：涉及完整账号、银行卡号、交易明细、扣款协议、分享、导出、生成收款信息单时，先做 mock 身份验证。
6. **AI 不做权限判断**：AI 只负责理解意图和组织结果；可展示内容由权限策略和 mock 数据服务决定。
7. **不编造银行数据**：无法从 mock source data 确认的账户、开户行、联行号、交易、授权协议，一律返回无法确认和安全下一步。
8. **渐进式信任提示**：不要让隐私/审计提示干扰主任务；普通状态轻量提示，敏感动作再弹确认。

## Demo 数据边界
使用湖州银行作为演示语境，但所有数据必须是虚构或脱敏演示数据。

禁止使用：
- 真实个人姓名、手机号、身份证号、地址、设备信息。
- 真实银行卡号、账号、联行号与个人账户绑定关系。
- 真实交易记录、扣款协议、授权商户、工资/养老金/补贴记录。
- 真实银行 API、真实认证、真实短信、真实人脸数据。
- 任何 `.env`、token、密钥、cookie、凭证。

允许使用：
- Fictional 用户：如“林晨”。
- Fictional 尾号：如“8026”“3391”。
- Fictional 支行：如“湖州银行演示支行”。
- Fictional 交易、商户和授权 App。
- Mock Face ID / 支付密码 UI，仅作为演示状态。

## 信息敏感等级

| 等级 | 信息类型 | 示例 | 默认展示规则 |
|---|---|---|---|
| 低敏 | 公开业务说明、普通术语解释、公开流程指引 | 联行号是什么、挂失和冻结区别 | 可直接展示 |
| 中敏 | 与登录账户相关但不含完整标识的信息 | 开户行、联行号、账户尾号、消费汇总、交易总额 | 登录态可展示，必要时脱敏 |
| 高敏 | 可识别账户、交易或授权关系的完整信息 | 完整卡号、完整账号、完整交易明细、扣款协议详情、电子回单 | 二次验证后展示，默认隐藏 |
| 高风险操作 | 可能影响资金、账户状态、授权状态或外发数据的动作 | 导出、分享、生成收款信息单、进入挂失/冻结、取消授权 | 二次验证 + 明确确认；第一版只做官方流程指引 |

## 操作风险等级

| 操作 | 风险 | 策略 |
|---|---|---|
| 查看公开说明 | 低 | 直接展示 |
| 查看摘要 | 低到中 | 登录态展示，避免完整敏感字段 |
| 查看单个字段 | 中到高 | 根据字段敏感度决定是否验证 |
| 显示完整账号/卡号 | 高 | 二次验证 + 环境安全提示 |
| 复制单行字段 | 中到高 | 对账号/卡号等字段要求验证后复制 |
| 复制整卡信息 | 高 | 二次验证；卡片内敏感字段默认脱敏或需确认包含完整字段 |
| 分享/生成收款信息单 | 高 | 二次验证 + 明确确认；第一版只模拟 |
| 查看完整交易明细 | 高 | 二次验证 |
| 导出交易/回单 | 高 | 二次验证 + 审计提示；第一版只模拟入口 |
| 管理扣款协议 | 高 | 进入官方流程；不由 AI 直接取消 |
| 挂失/冻结/补卡 | 高风险 | 进入官方流程；AI 只解释和引导 |

## 身份验证触发规则

必须触发 mock 二次验证：
- 用户请求完整银行卡号。
- 用户请求完整账号。
- 用户请求生成或分享收款信息单。
- 用户请求查看完整交易明细。
- 用户请求导出交易、电子回单或流水证明。
- 用户请求查看完整扣款协议或授权关系。
- 用户进入挂失、冻结、解挂、补卡、取消授权等官方流程。

可不触发二次验证：
- 已登录状态下查询开户行和联行号，并且只展示账户尾号。
- 已登录状态下查看消费分类汇总。
- 查询银行术语解释、挂失/冻结区别、公开流程说明。

需要先选择再验证或展示：
- 多张银行卡下查询银行卡号。
- 多张银行卡下查询开户行/联行号。
- 多个账户可生成收款信息时。
- 多个同名收款人/交易对象时。
- 多笔交易都可能对应“这笔扣款”时。
- 多张银行卡都存在自动扣款协议时。

## 多对象选择规则
当结果可能对应多个对象时，助手必须先弹出选择，不得默认替用户选择。

选择弹窗应包含：
- 类型：银行卡、账户、交易、扣款协议或收款人。
- 可安全展示的识别信息：银行名、卡类型、尾号、日期、金额、商户名。
- 不显示完整卡号、完整账号、完整交易对手敏感信息。
- 明确动作按钮：选择、取消、换一张卡、缩小范围。

示例：
- “请选择要查询的银行卡：湖州银行借记卡 尾号 8026 / 湖州银行工资卡 尾号 3391。”
- “我找到 3 笔可能相关的扣款，请选择要解释哪一笔。”

## 结果卡片规则
结果卡片必须短、清楚、可操作。

必须遵守：
- 先回答用户问的内容。
- 默认不展示用户没问的附加字段。
- 字段数量优先控制在 3 到 7 项。
- 对敏感字段默认脱敏。
- 每个可复制字段提供单行复制。
- 整卡复制只复制当前卡片可见字段；若包含完整账号/卡号，需要额外确认。
- “查看来源”“展开详情”“安全说明”作为次级操作，不占主视觉。

不应出现：
- 大段免责声明占据卡片底部。
- 把网点电话、地址、适用场景、说明文案全部堆到开户行卡片里。
- 用户只问卡号时，同时展示开户行、限额、余额、网点等无关信息。
- 用户只问单笔扣款时，默认展示整月消费报告。

## 渐进式信任提示
隐私和审计提示要存在，但不应破坏主要体验。

推荐层级：
1. **普通状态**：卡片右下角或底部一行轻量来源，例如“来自账户资料，10:30 更新”。
2. **敏感字段操作**：用户点击显示完整账号/卡号时弹确认。
3. **外发或导出操作**：用户点击分享、导出、生成信息单时显示更强确认。
4. **用户主动查看**：提供“查看来源/安全说明”，展开后说明数据来源、更新时间、验证原因和审计提示。

示例文案：
- “完整卡号属于敏感信息，请确认周围环境安全。”
- “导出交易记录会生成包含敏感金融信息的文件。”
- “本次查看完整账号会记录在安全日志中。”

## AI 与数据服务边界
第一版 demo 可以用规则化 intent routing 和 mock 数据实现，但概念上必须分层：

- Intent layer：识别用户想查银行卡号、开户行、转账记录、扣款解释、自动扣款等。
- Policy layer：判断是否需要验证、是否需要先选择对象、哪些字段可展示。
- Data layer：从 mock source data 返回账户、网点、交易、授权协议。
- Presentation layer：把允许展示的数据组成短卡片。

AI/助手不能：
- 自己生成不存在的账号、联行号、支行、交易或扣款协议；如未来扩展跨境场景，也不得生成不存在的 SWIFT/BIC 或中间行信息。
- 绕过 policy layer 直接展示高敏信息。
- 在没有用户确认时默认选择银行卡或交易对象。
- 直接执行转账、挂失、冻结、取消授权、投诉或导出。
- 把用户没有问的金融建议、营销推荐或投资建议塞进结果。

当数据不存在或不确定：
- 说清楚“我无法从当前账户数据确认”。
- 给出安全下一步，例如“请进入官方账户详情页确认”或“联系官方客服”。
- 不给猜测性字段。

## 场景安全矩阵

| 场景 | 默认展示 | 需要验证 | 多对象选择 | 禁止行为 |
|---|---|---|---|---|
| 境内收款信息 | 户名、银行、账号尾号、开户行、联行号 | 完整账号、分享、生成信息单 | 多卡/多账户时 | 自动分享真实信息 |
| 银行卡号 | 卡尾号 | 完整卡号、复制完整卡号 | 多卡时 | 附带无关余额/限额 |
| 开户行/联行号 | 开户行、联行号、卡尾号 | 一般不需要；完整账号需要 | 多卡时 | 只返回菜单路径 |
| 对某人转账 | 汇总、最近交易摘要 | 完整明细、导出、回单 | 同名对象/多笔交易时 | 默认展示全部明细 |
| 消费总结 | 分类汇总、异常提示 | 查看具体敏感明细 | 多账户/多卡时 | 给投资建议 |
| 这笔扣款是什么 | 单笔扣款解释 | 查看协议详情、投诉/争议入口 | 未指定交易时 | 默认展开所有历史扣款 |
| 自动扣款 App | 授权摘要 | 协议详情、取消/管理入口 | 多卡时 | AI 直接取消授权 |
| 银行卡丢失 | 冻结/挂失区别和官方入口 | 进入真实流程前 | 多卡时 | AI 直接挂失或冻结 |

## 审计事件设计
第一版不需要真实审计系统，但原型应体现哪些操作会被记录。

建议 mock audit event 字段：
- event_id
- timestamp
- user_id: demo_user
- session_id
- action_type: reveal_full_card_number / reveal_full_account / export_transactions / generate_receipt_sheet / open_lost_card_flow
- resource_type: card / account / transaction / authorization
- resource_ref: demo id or masked label
- auth_method: mock_face_id / mock_pay_password
- result: success / cancelled

原型里不需要展示完整 audit log。只需在敏感确认弹窗中用轻量文案提示“本次查看会记录在安全日志中”。

## 真实银行名称使用规则
湖州银行只作为 demo 语境。

必须避免：
- 暗示已获得授权、合作或接入。
- 使用真实客户数据或真实截图。
- 使用真实 UI 商标素材作为官方背书。
- 声称功能已经在真实银行 App 中上线。

可以表达：
- “以湖州银行语境做概念演示。”
- “所有数据为演示数据。”
- “真实落地需要接入银行账户系统、交易系统、权限策略和审计系统。”

## 实现前检查清单
进入前端实现前，必须确认：

- 每个 demo 场景有明确 intent。
- 每个 intent 有风险等级。
- 每个 intent 定义是否需要二次验证。
- 多卡/多账户/多交易时有选择弹窗。
- 每张结果卡片只展示用户所问的必要字段。
- 敏感字段默认脱敏。
- 复制、显示完整、分享、导出、生成信息单有不同风险处理。
- 所有数据来自 mock source data。
- 查不到数据时有安全 fallback。
- 没有真实个人信息、真实账号、真实交易、真实凭证或真实密钥。

## Phase 3 结论
这个 demo 的安全策略不是把体验变复杂，而是让便利可被银行接受：

- 普通问题直接答。
- 敏感信息先验证。
- 多个可能对象先选择。
- 卡片只答所问。
- 高风险动作进官方流程。
- AI 不编造、不越权、不执行真实金融动作。
